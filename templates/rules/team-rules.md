# 团队通用规则

本文件定义所有 Agent 必须遵守的通用规则。各 Agent 的 CLAUDE.md 应引用本规则。

---

## 1. 指挥链规则

- 董事长（Chairman）在群聊中发布的信息，**只能由 CEO 接收并分派**
- 除非董事长明确指定了具体 Agent，其他 Agent 不得直接处理董事长的群聊指令
- 收到董事长指令后的正确流程：CEO 接收 → CEO 判断分派 → 对应 Agent 执行
- 如果董事长在私聊中直接给某 Agent 下达指令，该 Agent 可直接执行，但需同步告知 CEO

## 2. Subagent 规则

- **除 CEO 和 Audit 外**，所有 Agent 接到任务后必须通过 subagent（子代理）执行具体工作
- 主会话（主进程）不能被长时间阻塞，必须保持消息接收能力
- subagent 执行完毕后，主会话负责汇报结果和状态更新
- **判断标准**：预计执行超过 2 分钟的任务**必须**用 subagent，包括但不限于：
  - 代码编写、修改、审查
  - 文件搜索和分析
  - 方案设计和文档编写
  - 任何涉及多步工具调用的操作
- **可在主会话直接处理的**（无需 subagent）：
  - 回复一条消息（确认、应答、简短回答）
  - 查询一个状态（get_state、get_agents、list_tasks）
  - 转发或分派任务
- **违规后果**：主会话被长任务阻塞导致消息堆积、无法响应团队沟通，影响协作效率

## 3. Task 系统规则

- 所有任务必须通过 Task 系统全程管理，流程如下：
  1. **create_task** — 创建任务，明确标题、描述、指派人
  2. **update_task** — 开始执行时更新状态为 `doing`，过程中可更新进度描述
  3. **done_task** — 任务完成后标记为 `done`，附上完成说明
- 任务状态流转：`todo` → `doing` → `done`
- 禁止绕过 Task 系统直接执行任务，确保所有工作可追踪、可审计
- 任务粒度建议：一个 Task 对应一个可独立验收的交付物

## 4. GitHub Push 规则

- **默认不执行 git push**，所有代码变更仅保留在本地
- 只有在董事长**主动提出**推送请求时才执行 push
- CEO 无权批准 push，只有董事长有此权限
- push 前必须确认：
  - 代码已通过测试（C 验收通过）
  - 没有敏感信息（密钥、token、.env 等）泄露
  - commit message 规范清晰
- 禁止 force push 到 main/master 分支

## 5. State 系统规则

- 关键决策和状态变更必须同步写入 State 系统（通过 set_state 工具）
- 需要写入 State 的场景包括：
  - 项目阶段变更（如从开发转入测试）
  - 重要技术决策（如架构选型、方案变更）
  - 阻塞问题和解决方案
  - 里程碑完成记录
- State 的 key 命名规范：`项目名/类别/具体项`，如 `arc-agi-3/status/phase`
- 其他 Agent 可通过 get_state / subscribe_state 获取最新状态，减少重复沟通

## 6. 沟通规范

- **群聊**：使用 `send_message` 工具，发送到对应频道
  - 项目相关讨论发到项目频道
  - 全员通知发到 #general
- **私聊**：使用 `send_dm` 工具，用于一对一沟通
  - 任务分配、进度催促、敏感问题使用私聊
- **沟通原则**：
  - 任务完成后主动汇报，不等催促
  - 遇到阻塞及时上报，不自行搁置
  - 跨角色协调通过 PM 或在公共频道进行，避免私下绕过流程

## 7. 消息响应规范

- **不是每条消息都需要回应**，只回应与自己直接相关的消息：
  - 被 @ 提及的消息
  - 私聊（DM）消息
  - 与自己职责直接相关的讨论
- **不要重复别人已说过的内容** — 如果其他 Agent 已经充分回答，无需再说一遍
- **点名/签到类消息除外** — 董事长要求点名时所有人应回应
- **目的**：避免消息洪峰导致 API rate limit，保持系统稳定

## 8. 消息内容规范

- **群聊/私聊只发结论和关键信息**，禁止以下内容：
  - 完整的工具输出（Bash 执行结果、文件内容等）
  - 完整的代码 diff 或大段代码
  - 工具调用的原始 JSON 或日志
- **异常信息**：只贴关键错误片段（如错误消息、堆栈第一行），不贴完整日志
- **代码引用**：说明文件名和行号即可（如 `router.mjs:150`），不贴代码块，除非对方明确要求
- **群聊消息长度上限：30 行**。超过 30 行的内容必须：
  - 用 `set_state` 存储完整内容，消息中放摘要 + state key
  - 或写入文件，消息中放摘要 + 文件路径
  - 或拆分为多条消息（每条不超过 30 行）
- **私聊可适当放宽**，但同样应精简，避免无意义的冗长

## 9. Server 操作权限

- **Server 启停** — 仅限 CEO / HR 执行，其他 agent 禁止自行启动或重启 server
- **启动必须带 `TEAMMCP_HOME` 环境变量** — 缺少此变量会导致读取错误数据库
- **DB 操作** — 禁止直接操作数据库文件，所有数据变更通过 API 进行

## 10. Dev/Prod 双环境规范

- **生产环境**（:3100）— 日常工作、正式任务、正式沟通，禁止破坏性测试
- **开发环境**（:3200）— 代码验证、新 agent 调试、破坏性测试、新功能验证

| | 生产 (Prod) | 开发 (Dev) |
|--|------------|-----------|
| 端口 | :3100 | :3200 |
| Agent 目录 | `Desktop/agents/` | `Desktop/agents-dev/` |
| 数据目录 | `Desktop/teammcp/` | `Desktop/teammcp-dev/` |
| 启动脚本 | `start-prod.ps1` | `start-dev.ps1` |

- **代码改动流程**：修改 server 代码 → dev 重启验证 → dev 环境测试通过 → CEO 协调重启 prod
- **新 agent 流程**：dev 注册和测试 → 稳定后迁移到 prod
- **禁止在 prod 环境直接测试未验证的代码改动**
